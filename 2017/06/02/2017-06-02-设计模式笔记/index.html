设计模式学习笔记

发表于 2017-06-02 | |

java开发中的23种设计模式学习笔记。

  • 一、设计模式的分类

    总的来说，设计模式可分为三大类，分别为：
  1. 创建行模式：工厂方法模式，抽象工厂模式，单例模式，建造者模式，圆形模式。
  2. 结构性模式：适配器模式，装饰器模式，代理模式，外观模式，桥接模式，组合模式，享元模式。
  3. 行为性模式：策略模式，模板方法模式，观察者模式，迭代子模式，责任链模式，命令模式，备忘录模式，状态模式，访问者模式，终结者模式，解释器模式。
    另外还有两类：并发型模式和线程池模式。

  • 二、设计模式六大原则

  1. 开闭原则：对扩展开放，对修改关闭。意思就是当程序需要新功能时不能改原有代码，而是可以新增方法或接口，易于维护和升级。
  2. 里氏代换原则：任何基类可以出现的地方，其子类都可以出现，也就是说子类完全继承实现了基类的一切，这样才能保证基类的复用，子类也保证了可以在基类基础上进行扩展而不影响功能。
  3. 依赖倒转原则： 面向接口编程思想，也就是说系统的逻辑和架构通过抽象类或接口进行定义和规范，而不是在具体实现类当中进行描述。
  4. 接口隔离原则： 尽量减少类依赖的接口中的方法，也就是说尽量将依赖的接口拆分，其中尽量不包含对类不必要的方法。
  5. 迪米特法则： 又称最少知识原则，就是解耦，一个类其功能的实现逻辑尽量不依赖外部类，而是封装在类内部，以减少外部类对其的影响。
  6. 单一职责原则： 尽量一个类或方法实现单一的业务或操作，这样可以解耦，避免了修改代码时可能引起的问题，并且颗粒度小，方便复用和修改。

  • 三、设计模式详解：

    • 工厂方法模式：
      • 普通工厂模式：将要生产的实例抽象出其抽象类，由Factory根据输入条件进行生产并返回。如对于用户对象，抽象出一个User，然后若干实例如用户1、用户2都继承User，Factory里有一个类似getUser的方法，根据输入的参数确定具体返回User1或User2，方法本身返回值类型就是User。
      • 多个工厂模式：为避免参数无效产生的错误或无法获取等问题，将获取不同实例的方法拆分成各个独立方法，如getUser1(),getUser2()等。
      • 静态工厂方法模式：将上面多个工厂模式的获取方法写成静态方法，这样就可以直接调用，不需要实例化工厂。
    1. 抽象工厂模式：
      工厂模式由于类的创建依赖于工厂这个实体类，导致如果要拓展程序，就需要修改工厂类代码，违背闭包原则。抽象工厂模式可以解决这个问题，将工厂也抽象出来，这样拓展时就不用改原来的工厂类，而是通过新加工厂类和生产的实例关系就可以了。
    2. 单例模式：保证了在java应用中jvm中只有一个实例存在，对于一些复杂类而言可以节省实例化的开销，同时也减少了GC压力，最重要的应用场景就是该实例在系统中本身具有特殊意义，不可以被创建导致注入权限、关键信息被篡改等。当然，在单例创建时需要考虑线程安全和并发等问题。
    3. 建造者模式：与工厂模式类似，区别在于工厂模式只返回实例对象，而建造者模式可以返回对象的组合，比如说list

    • 对象的适配器模式： 与类的适配器思路类似，不过adapter不是继承目标类，而是持有目标类的实例。

    • 接口的适配器模式：由于实现接口需要实现所有的方法，而这样很有可能造成浪费，因为该类不需要这么多方法，所以就需要接口的适配器，实现思路是通过一个抽象类，该抽象类实现该接口，实现所有方法，然后我们的类继承该抽象类就可以不必重写所有方法，只重写需要的方法即可。

  • 装饰模式：就是给一个对象增加一些新的功能，而且是动态的，就是不在目标类目标方法修改，而是通过装饰类加一个方法包含被装饰的方法，例如：

    在实例化一个source目标类之后，在调用装饰类的构造方法添加被装饰的类，之后调用装饰类的装饰方法就可以实现装饰效果。

  • 代理模式：实现方式类似于装饰模式，作用是用于改进原有方法而不修改原代码。
  • 外观模式：将类和类之间的关系放到一个Facade类中，以降低类之间的耦合度。
  • 桥接模式：将抽象化与实现化解耦，使得二者可以独立变化。其实就是将要实现的功能抽象到一个接口，然后将该接口交给Bridge，在Bridge的实现类当中进行调用

  • 享元模式：主要目的是实现对象的共享，当系统中对象比较多的时候可以减少内存的开销，通常与工厂模式一起使用。享元工厂中持有一个对象池，当客户端发出请求时，工厂检验是够存在符合条件的对象，如果有则返回该对象，如果没有则新建一个对象返回，类似于多例模式。
